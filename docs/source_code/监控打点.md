# 监控打点

feapder 内置打点监控基于 **InfluxDB 2.x**，并提供 [Grafana 看板模板](报警及监控?id=可视化监控)。

## 一、依赖与部署

需要一个 InfluxDB 2.x（>= 2.7）实例。最简的本地部署：

```bash
docker run -d --name influxdb2 \
  -p 8086:8086 \
  -e DOCKER_INFLUXDB_INIT_MODE=setup \
  -e DOCKER_INFLUXDB_INIT_USERNAME=admin \
  -e DOCKER_INFLUXDB_INIT_PASSWORD=adminpass \
  -e DOCKER_INFLUXDB_INIT_ORG=feapder \
  -e DOCKER_INFLUXDB_INIT_BUCKET=feapder \
  -e DOCKER_INFLUXDB_INIT_ADMIN_TOKEN=my-token \
  influxdb:2.7
```

启动完成后，登录 `http://localhost:8086` 即可看到默认创建的 org `feapder` 与 bucket `feapder`，token 为 `my-token`。生产环境建议在 InfluxDB 中单独创建一个仅对该 bucket 有写权限的 token。

> 进程级 runtime 指标依赖 [`psutil`](https://pypi.org/project/psutil/)，已在 `feapder/requirements.txt` 中声明，正常安装 feapder 即包含。

## 二、配置

打点功能默认关闭，需在 `setting.py` 中显式启用并配置 InfluxDB 连接信息：

```python
INFLUXDB_ENABLE = True                   # 必填：默认 False，显式置 True 才启用
INFLUXDB_URL = "http://localhost:8086"
INFLUXDB_TOKEN = "my-token"
INFLUXDB_ORG = "feapder"
INFLUXDB_BUCKET = "feapder"
```

只要 `INFLUXDB_ENABLE = True` 且 `url / token / org / bucket` 都配置完整，框架会在爬虫启动时自动初始化打点客户端。

可选的高级参数：

```python
METRICS_DEFAULT_TAGS = {"env": "prod"}   # 公共 tag，会附加到每一个数据点上
METRICS_ADD_HOSTNAME = False             # 是否将 hostname 作为 tag
METRICS_EMIT_INTERVAL = 10               # 后台 flush 间隔（秒）
METRICS_BATCH_SIZE = 500                 # 单批写入最大点数
METRICS_RUNTIME_INTERVAL = 5             # 进程级 runtime 指标采样间隔（秒），<=0 关闭
METRICS_DEBUG = False                    # 是否打印调试日志
METRICS_OTHER_ARGS = dict()              # 透传给 MetricsEmitter 的其他参数
```

写入路径直接复用 InfluxDB 官方客户端的非同步批量模式（`WriteOptions`），失败自动重试 3 次，业务线程不会被网络阻塞。

## 三、内置打点

框架在以下位置已经内置打点，开箱即用：

| Measurement | 来源 | Tags | 说明 |
| --- | --- | --- | --- |
| `feapder_download` | `parser_control` | `spider`, `parser`, `status` | 下载/解析事件 |
| `feapder_item` | `item_buffer` | `spider`, `table` | 入库事件，仅按表统计总条数 |
| `feapder_proxy` | `proxy_pool` | `spider`, `event` | 代理池事件 |
| `feapder_user` | `gold_user_pool` | `spider`, `status` | 账号池事件，按状态聚合 |
| `feapder_runtime` | 后台采样线程 | `spider` | 进程级指标，fields: `cpu_percent` / `memory_rss_mb` / `num_threads` |
| `feapder_custom` | 业务自定义 | `spider`, `metric`, ...自定义 | `emit_custom` 写入的数据 |

`feapder_download.status` 取值：

| 值 | 含义 |
| --- | --- |
| `download_total` | 进入下载流程的请求数 |
| `download_success` | 下载并解析成功 |
| `download_exception` | 网络/HTTP 层错误（连接失败、超时、SSL 等） |
| `parser_exception` | 解析回调里抛异常 |

`feapder_proxy.event` 取值：`pull` / `use` / `invalid`。

> 同一 measurement + tags 在同一秒内的 counter 会被自动累加为一个点，避免 InfluxDB 写压力放大；如需更细的时间粒度请使用 `emit_custom(..., duration=...)` 或 `value=...`。

## 四、自定义打点

框架在 `feapder.utils.metrics` 中导出了一组语义化 API。AirSpider 与 Spider 启动时会自动调用 `metrics.init()`，业务代码直接调用即可：

```python
import feapder
from feapder.utils import metrics


class TestSpider(feapder.AirSpider):
    def start_requests(self):
        yield feapder.Request("https://www.baidu.com")

    def parse(self, request, response):
        # 计数类指标（按秒聚合）
        metrics.emit_custom("parse_success", count=1)

        # 数值类指标
        metrics.emit_custom("queue_size", value=128)

        # 时长类指标
        metrics.emit_custom("render_time", duration=1.23)


if __name__ == "__main__":
    TestSpider().start()
```

如果在 scrapy 或独立 Python 脚本中使用，需要手动调用 `metrics.init()` 与 `metrics.close()`：

```python
from feapder.utils import metrics

metrics.init(
    url="http://localhost:8086",
    token="my-token",
    org="feapder",
    bucket="feapder",
    spider="my_script",
)
metrics.emit_custom("api_call", count=1, tags={"endpoint": "/login"})
metrics.close()
```

> 即使忘记 `close()`，框架也会通过 `atexit` 钩子在进程退出时自动 flush 一次缓冲；fork 出的子进程会被自动重置，下次 emit 时按 pid 重新初始化。

## 五、API 一览

```python
# 下载/解析事件
metrics.emit_download(spider, parser, status, count=1)

# 入库事件（按表）
metrics.emit_item(spider, table, count=1)

# 代理池事件
metrics.emit_proxy(spider, event, count=1)

# 账号池事件（按状态）
metrics.emit_user(spider, status, count=1)

# 自定义事件，count / value / duration 三选一
metrics.emit_custom(metric, *, count=None, value=None, duration=None,
                    spider=None, tags=None, measurement="feapder_custom")
```

每个同步函数都对应一个协程版本：`aemit_download` / `aemit_item` / `aemit_proxy` / `aemit_user` / `aemit_custom`，内部基于 `asyncio.to_thread`，可与 `async def parse` 协作。

> 当 `metrics.init` 时已传入 `spider` 参数（框架内部会自动传入爬虫名），后续所有 `emit_*` 调用均可省略 `spider` 参数，会自动以 `init` 时设置的值兜底。

### 高基数 tag 注意事项

InfluxDB 的 tag 是索引化的，高基数 tag 会让 series 暴涨、查询与内存性能急剧下降。请遵循：

- `emit_custom` 的 `metric` 与 `tags` 中**不要**写 `trace_id`、`request_id`、`user_id` 等高基数值；改用 `value=` 把它存成 field，或 `duration=` 走 timer 类型。
- 框架内置的 `feapder_user` 仅按 `status` 维度聚合，不再支持按账号 ID 下钻。

## 六、与旧版本的破坏性变更

| 变更 | 说明 | 迁移 |
| --- | --- | --- |
| `emit_user` 去掉 `user_id` 参数 | 账号池打点高基数风险，仅保留 `spider`、`status` 维度 | 删除调用处的 `user_id=` 参数 |
| `emit_item` 去掉 `field` 参数 | 字段维度打点对 InfluxDB 写压力放大严重，且实际使用场景少 | 删除调用处的 `field=` 参数；自定义字段维度需求改用 `emit_custom(metric="字段填充率", tags={...})` 自行实现 |
| 写入路径切换为非同步批量 | 由 InfluxDB 官方 `WriteOptions` 接管批量与重试 | 业务无感知；自定义透传给 `MetricsEmitter` 的 `max_timer_seq`、`max_points` 等老参数会被忽略 |
